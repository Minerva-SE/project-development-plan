\section{项目概述}

\subsection{工作内容}

\subsubsection{项目的规模}

\begin{enumerate}[(1)]
  \item \textbf{系统功能范围}\\
  本系统——Minerva：全链路的轻量级 EEG 数据管理平台——主要涵盖以下功能模块，在需求分析书中有具体介绍：
    \begin{itemize}
      \item \textbf{用户与权限管理模块}：三类角色（被试者、实验员、管理员）的注册、登录、审批与权限控制。
      \item \textbf{实验项目管理模块}：实验项目的创建、编辑、状态流转与归档管理。
      \item \textbf{EEG 数据管理模块}：原始数据上传、元数据维护、标签分类与全生命周期状态管理。
      \item \textbf{自动解析与质控模块}：文件格式解析、通道完整性检查、异常值检测与质控报告生成。
      \item \textbf{伦理审核与授权模块}：知情同意书电子化签署、数据使用申请与审批、授权撤回。
      \item \textbf{数据分析模块}：时域波形、频谱分析、频段能量计算与 AI 状态识别。
      \item \textbf{通知与 Dashboard 模块}：角色化首页仪表盘与站内消息通知。
      \item \textbf{系统配置与审计模块}：运行参数配置与全链路操作日志追溯。
    \end{itemize}

  \item \textbf{预期用户规模}\\
  本系统定位于教学与基础科研场景，用户规模以课程教学班级和校内基础科研团队为参照：
    \begin{itemize}
      \item \textbf{被试者}：系统上线初期预计服务于 1--2 个教学班级，约 40 名被试者角色用户（以学生模拟扮演为主）。随着课程推广，预计一学期内被试者用户可达 100 人以上。
      \item \textbf{实验员}：预计初期有 10--20 名实验员（承担课程实验的学生及基础科研人员），一学期内可增长至 50 人以上。
      \item \textbf{管理员}：预计设置 2--3 名系统管理员（项目开发者），负责用户审批、数据审核和系统配置维护。
    \end{itemize}

  \item \textbf{数据量预估}
    \begin{itemize}
      \item \textbf{实验项目数据}：系统上线初期预计录入 10--20 个实验项目（覆盖 1--2 门课程的实验教学需求），一学期内预计增长至 50 个实验项目以上。
      \item \textbf{EEG 数据记录}：按每个实验项目包含 5--10 条 EEG 数据记录估算，初期数据量约 50--100 条；一学期后预计达到 200 条以上。
      \item \textbf{用户数据}：包括被试者、实验员和管理员三类角色，初期约 50 条，预计系统上线一学期后增长至 100 条以上。
      \item \textbf{EEG 原始文件存储}：按单文件平均 50 MB 估算，初期总存储量约 5 GB，一学期后预计达到 25 GB 以上。
    \end{itemize}

  \item \textbf{开发工作量}\\
  项目整体开发周期约 8 周，按敏捷迭代模式组织：
    \begin{itemize}
      \item \textbf{产品经理}：1 人，全程参与，负责需求分析、原型设计、进度跟踪与团队协调。
      \item \textbf{前端开发}：1 人/周期，负责 React + TypeScript 界面实现与可视化渲染。
      \item \textbf{后端开发}：3 人/周期，负责 Django REST API、数据库设计与核心业务逻辑。
      \item \textbf{数据开发}：2 人/周期，负责 EEG 数据解析、质控算法与 AI 分析模块。
      \item \textbf{测试}：2 人/周期，负责测试用例设计、功能测试、集成测试与缺陷跟踪。
      \item \textbf{文档与配置管理}：由产品经理和测试人员共同承担。
    \end{itemize}
    开发采用三轮迭代，每轮迭代约 2 周，加上前期的准备阶段（约 1 周）和最终的验收交付阶段（约 1 周），共需约 8 周。
\end{enumerate}

\subsubsection{项目的软件生命周期及模型}

本项目选取敏捷开发模型作为软件生命周期模型，主要基于以下考虑：

\begin{itemize}
  \item \textbf{需求的可探索性}：EEG 数据管理与科研伦理审核是一个领域性较强的课题，团队成员在项目初期对具体功能边界的理解可能存在偏差。敏捷模型允许在迭代过程中根据阶段性评审反馈调整需求优先级和实现方案，确保最终交付物贴合实际教学科研场景。
  \item \textbf{快速交付与持续反馈}：敏捷开发强调在短周期内交付可运行的软件增量。本项目划分为三轮迭代，每轮迭代末均安排演示与评审，便于指导教师和团队成员及时发现设计偏差与实现问题，在下一轮迭代中予以纠正。
  \item \textbf{团队协作与沟通}：项目团队成员分属不同角色（产品、前端、后端、数据、测试），敏捷模型中每日站会、迭代计划会和回顾会等机制有助于打破角色壁垒，确保信息透明和任务对齐。
  \item \textbf{风险早期暴露}：EEG 数据处理涉及文件解析、异步任务调度和信号分析算法等技术难点。敏捷迭代模式允许在早期迭代中优先验证关键技术路径（如文件解析流水线、认证授权链路），降低后期集成阶段的技术风险。
\end{itemize}

项目的主要阶段为：需求与分析 $\to$ 计划与设计 $\to$ 迭代开发（每轮含计划、开发、测试、评审）$\to$ 验收与交付。具体的软件生命周期及迭代安排详见第 4 章"工作分解"。

\subsubsection{项目的软件功能}

为了满足 EEG 数据管理与科研伦理审核的要求，Minerva 系统的功能模型按三类用户角色组织如下：

\begin{enumerate}[(1)]
  \item \textbf{基础信息与用户管理模块}\\
    \textbf{面向所有角色：}
    \begin{itemize}
      \item 注册与登录：用户在注册界面输入手机号、密码和角色（被试者/实验员），通过短信验证码校验后完成注册；已注册用户通过账号密码登录，系统签发 JWT Token 并根据角色路由至对应工作台。新注册的实验员账号需等待管理员审批后方可激活。
      \item 查看/修改个人信息：用户登录后可查看和编辑个人资料（姓名、邮箱、手机号等），支持密码修改。
      \item Dashboard 首页面板：登录后默认展示角色化仪表盘——管理员展示待办审批量和系统概况，实验员展示个人项目进度和上传数据量，被试者展示授权状态提醒。
    \end{itemize}
    \textbf{面向管理员：}
    \begin{itemize}
      \item 管理审核用户信息：管理员可查看所有注册用户列表，对待审核账号进行审批（通过/驳回），对已激活账号进行禁用或恢复操作。
    \end{itemize}

  \item \textbf{EEG 实验与数据管理模块}\\
    \textbf{面向实验员：}
    \begin{itemize}
      \item 创建与管理实验项目：实验员创建新实验，录入名称、编号、用途类型（教学/科研）、描述和时间范围等元数据；在实验生命周期内维护其状态（草稿 $\to$ 进行中 $\to$ 已结束 $\to$ 已归档）。
      \item 上传 EEG 数据文件：在实验下上传原始 EEG 文件（支持 CSV、EDF 等格式），绑定被试者和知情同意记录，填写采样率、通道配置等元数据，添加自定义标签。
      \item 查看数据详情与质控报告：在数据列表中查看各记录的处理状态和审核状态；进入详情页查看质控报告——包括格式解析结果、缺失值比例、异常值统计、通道完整性和综合评分。
      \item 导出与下载：经授权的数据支持下载原始文件和派生数据。
    \end{itemize}
    \textbf{面向管理员：}
    \begin{itemize}
      \item 审核 EEG 数据：管理员查看待审核数据及其质控报告，做出通过、驳回或冻结的决定，填写审核意见。
      \item 管理数据标签：维护系统内数据标签的分类体系。
    \end{itemize}

  \item \textbf{伦理审核与授权管理模块}\\
    \textbf{面向被试者：}
    \begin{itemize}
      \item 签署知情同意书：在线阅读当前生效版本的知情同意书模板，理解数据采集目的、使用范围和隐私保护措施后完成电子化签署；对已签署的同意书有权撤回。
      \item 查看与管理授权：查看与本人相关的数据使用授权记录（授权对象、使用目的、时间范围），对有效授权提交撤回申请。
    \end{itemize}
    \textbf{面向实验员：}
    \begin{itemize}
      \item 提交数据使用申请：对质控通过的数据提交使用申请，说明使用目的、所需访问范围（查看/下载/分析）和期望的导出格式。
    \end{itemize}
    \textbf{面向管理员：}
    \begin{itemize}
      \item 审批数据使用申请：管理员查看申请详情和关联的知情同意签署状态，做出通过或驳回决定，指定授权的访问范围和有效期限。
      \item 管理知情同意书模板：创建和维护知情同意书模板，管理模板版本——系统同时仅允许一个 Active 版本生效，历史版本保留备查。
      \item 查看操作审计日志：按操作人、操作类型、对象类型和时间范围筛选和追溯系统操作记录。
    \end{itemize}

  \item \textbf{数据分析与 AI 识别模块}\\
    \textbf{面向实验员：}
    \begin{itemize}
      \item 时域波形可视化：选择 EEG 数据和目标通道，查看多通道波形叠加显示。
      \item 统计分析：计算均值、方差、峰值等统计指标。
      \item FFT 频谱分析：对选定通道和时间段执行快速傅里叶变换，展示频谱图。
      \item 频段能量分析：计算 $\alpha$、$\beta$、$\delta$、$\theta$、$\gamma$ 等频段的能量分布。
      \item AI 状态识别：基于预训练模型对 EEG 数据进行状态分类（如睁眼/闭眼、注意力水平等），返回分类结果和置信度。
    \end{itemize}

  \item \textbf{系统配置模块}\\
    \textbf{面向管理员：}
    \begin{itemize}
      \item 维护系统运行参数：配置文件上传大小上限（默认 200 MB）、支持的文件格式列表、默认采样率和通道数、Token 有效期、验证码有效期、日志保留时长和通知开关等。
    \end{itemize}
\end{enumerate}

\subsubsection{项目的软件性能}

系统的性能从以下方面来衡量，作为后续测试的指标：

\begin{enumerate}[(1)]
  \item \textbf{界面设计}\\
  系统 UI 应简洁清晰，布局合理，突出数据管理流程的核心操作。导航结构按角色区分，被试者侧重授权管理、实验员侧重数据操作、管理员侧重审批与监控。支持浅色/深色主题切换，并在刷新后保持用户选择。

  \item \textbf{响应速度}\\
  系统应具有良好的响应速度，给用户良好的使用体验。在校园网或普通宽带环境下，系统应满足以下时间特性要求：
    \begin{itemize}
      \item 首屏加载：前端生产构建后的首页和登录页应在 3 秒内完成主要内容渲染。
      \item 页面切换：已加载应用内的导航切换、列表页到详情页跳转应在 1 秒内完成主要内容更新。
      \item 接口响应：常规查询接口平均响应时间不超过 1 秒；审批、上传、分析任务创建等写操作应在 3 秒内返回提交结果或任务状态。
      \item 文件上传：单个 EEG 文件上传时，前端应显示上传进度；文件大小超限或格式不支持时应给出明确提示。
      \item EEG 波形渲染：多通道波形图在常规通道数（32 通道以内）下的初次渲染应在 3 秒内完成。
    \end{itemize}

  \item \textbf{访问容量}
    \begin{itemize}
      \item 并发支持：课程演示环境下应支持至少 20 名用户同时登录并执行查询、上传、申请和审批等常规操作。
      \item 列表规模：在 100 条以内实验、数据、申请或日志记录的常规课程数据规模下，列表渲染、搜索和筛选不应出现明显卡顿。
    \end{itemize}

  \item \textbf{服务器配置最低要求}
    \begin{itemize}
      \item CPU：2 核及以上。
      \item 内存：4 GB。
      \item 磁盘：40 GB（根据 EEG 文件存储需求扩展）。
      \item 操作系统：Ubuntu 22.04 LTS 或兼容 Linux 发行版；开发环境可使用 macOS 或 Windows。
    \end{itemize}

  \item \textbf{数据处理能力}\\
      系统应能够处理课程教学和基础科研场景下的 EEG 数据管理需求，满足以下数据容量要求：
    \begin{itemize}
      \item 至少支持 300 条用户数据。
      \item 至少支持 50 条实验项目数据。
      \item 至少支持 500 条 EEG 数据记录及其关联的质控报告、处理结果和派生数据。
      \item 至少支持 500 条知情同意记录和 500 条授权记录。
      \item 至少支持 5000 条操作日志数据。
      \item 单文件上传上限默认 200 MB，支持 CSV、EDF 等常见 EEG 文件格式。
    \end{itemize}

  \item \textbf{可用性}
    \begin{itemize}
      \item 多浏览器支持：系统应在 Chrome、Edge、Firefox 等主流浏览器的近两年稳定版本中正确显示和运行。
      \item 设备兼容：桌面端和笔记本电脑为主要使用设备；平板设备可用于浏览和轻量审批；手机端不作为主要数据分析终端。推荐分辨率宽度 1280px 及以上。
      \item 异常提示：在数据无权限、账号待审核、请求失败或处理失败时应展示明确提示，避免用户进入错误流程。
    \end{itemize}
\end{enumerate}

\subsection{条件和限制}

\subsubsection{条件}

\begin{enumerate}[(1)]
  \item \textbf{技术条件}
    \begin{itemize}
      \item \textbf{开发环境}：项目开发团队需具备稳定的开发环境，包括代码编辑器（VS Code）、版本控制系统（Git/GitHub）、Node.js 20 LTS 及以上（前端）、Python 3.11 及以上（后端）、PostgreSQL 14 及以上（数据库）。所有开发人员需熟悉项目技术栈：前端 Vite + React + TypeScript，后端 Django REST Framework，数据库 SQL。
      \item \textbf{网络安全}：系统涉及模拟的个人可识别信息（PII）和生物特征数据（EEG 信号），必须确保数据安全。开发团队需遵循安全最佳实践——前端使用 Bearer Token 进行接口认证，后端所有受保护接口校验用户角色与数据权限，关键操作写入不可篡改的审计日志。演示环境建议使用 HTTPS；本地开发可使用 HTTP。敏感配置（数据库口令、Token 密钥等）不得写入前端代码或公开文档。
      \item \textbf{数据处理流水线}：EEG 文件的解析和质控需要异步任务支持，后端应具备任务队列管理能力，确保文件处理不阻塞主请求线程。
    \end{itemize}

  \item \textbf{人员条件}
    \begin{itemize}
      \item \textbf{人员稳定性}：项目开发周期约 8 周（4 月 27 日至 6 月 22 日），团队共 9 人（产品经理 1 人、前端开发 1 人、后端开发 3 人、数据开发 2 人、测试 2 人）。所有成员均为浙江大学本科生，需同时兼顾课程学习和其他任务，团队成员的相对稳定和合理分工至关重要。
      \item \textbf{讨论与学习}：项目涉及 EEG 数据管理领域知识和科研伦理规范，团队成员需具备快速学习的能力。团队定期召开会议进行进度同步和技术交流，确保所有成员对业务需求和技术方案的理解保持一致。
      \item \textbf{协作能力}：团队成员分属不同专业方向，需通过 GitHub Pull Request 代码评审、飞书文档协作和定期站会等方式保持高效沟通。
    \end{itemize}
\end{enumerate}

\subsubsection{限制}

\begin{enumerate}[(1)]
  \item \textbf{时间限制}\\
  项目开发周期为 8 周，团队须在以下里程碑节点前完成相应的工作内容，并进行阶段性评审和验收。

  \item \textbf{法律和政策限制}
    \begin{itemize}
      \item \textbf{数据保护法规}：系统涉及模拟的个人生物特征数据（EEG 信号），须遵循《中华人民共和国数据安全法》《中华人民共和国个人信息保护法》的基本原则。开发过程中须使用模拟数据集，禁止使用真实被试数据；数据模型设计中须体现数据脱敏和隐私保护的机制。
      \item \textbf{科研伦理规范}：系统内伦理审核流程的设计须参照《赫尔辛基宣言》和 ICH-GCP 的基本原则，体现知情同意、隐私保护和数据最小化使用的核心理念。系统内的"伦理审核"为教学模拟流程，不替代真实伦理委员会的法律审查职能。
      \item \textbf{知识产权保护}：项目开发过程中所使用的第三方库和工具须为开源许可或合法授权。团队须保护自身开发的代码和文档的知识产权，遵守浙江大学课程作业的学术诚信规范。
    \end{itemize}

  \item \textbf{市场和用户限制}
    \begin{itemize}
      \item \textbf{用户需求变化}：作为课程项目，系统的核心需求在需求分析阶段已基本确定，但不排除在迭代评审中根据指导教师反馈进行调整。团队须在敏捷框架下管理需求变更，确保核心功能在既定时间范围内完成。
      \item \textbf{领域竞争}：国际上已有 EEGLAB、MNE-Python 等成熟的 EEG 分析工具和 OpenNeuro 等数据共享平台。本系统的差异化定位在于填补"教学与基础科研场景下的数据管理与伦理审核系统"这一细分空白，不追求算法的极致丰富性，而是聚焦数据治理闭环和伦理合规流程。
    \end{itemize}
\end{enumerate}

\subsection{产品}

\subsubsection{程序}

Minerva：全链路的轻量级 EEG 数据管理平台。

\subsubsection{文档}

\begin{enumerate}[(1)]
  \item \textbf{会议记录}
  项目开发过程中每两周进行一次全体会议讨论，进行会议记录。如有临时会议也需要进行会议记录。格式为《yyyy-mm-dd.md》，存放于项目文档仓库。

  \item \textbf{迭代记录}
  项目开发过程中每轮迭代（Sprint 1--3）结束时进行迭代评审和回顾，迭代记录提供本次迭代已完成的任务及下次迭代需要完成的需求。格式为《第 x 轮迭代.md》，附于项目总结文档中。

  \item \textbf{过程文档}
  为课程要求，主要包括《需求分析书》、《项目开发计划书》、《项目总结报告》等文档，存放于项目文档仓库。
\end{enumerate}

\subsubsection{运行环境}

\begin{enumerate}[(1)]
  \item \textbf{操作系统}
    \begin{itemize}
      \item 服务端：Ubuntu 22.04 LTS 或兼容 Linux 发行版（生产环境推荐）；开发环境可使用 macOS 或 Windows。
      \item 客户端：Windows 11、macOS 13.0.1 及以上、Ubuntu 22.04 等主流操作系统。
    \end{itemize}

  \item \textbf{设备要求}
    \begin{itemize}
      \item 浏览器要求：Chrome、Edge、Firefox 等主流浏览器的近两年稳定版本。
      \item 分辨率要求：推荐宽度 1280px 及以上；系统应在 760px 以上宽度保持主要功能可用。
      \item 网络要求：能够稳定访问前端站点和后端 API；上传 EEG 文件时应保持连接不中断。
      \item 客户端权限：浏览器需允许 JavaScript、LocalStorage 和文件选择能力，以支持主题保存、登录状态和数据上传。
    \end{itemize}

  \item \textbf{前端依赖}
    \begin{lstlisting}[language=json]

  "dependencies": {

    "vite": "^7.1.9"
  },
  "devDependencies": {
    "@types/react": "^19.2.14",

    前端部署时至少应配置以下运行参数：
    \begin{itemize}
      \item 后端 API 基础 URL 地址。
      \item JWT Token 存储方式（如 LocalStorage）和过期处理策略。
      \item 文件上传大小上限和允许的文件格式列表。
      \item 默认主题（浅色/深色）和主题切换持久化策略。
      \item 跨域访问配置（如 CORS 或同源反向代理规则）。

      \item Token 密钥、访问 Token 有效期和刷新策略。
      \item EEG 文件上传目录、单文件大小上限和允许文件格式。
      \item 日志保留周期、审计日志写入策略和异常告警方式。
      \item 跨域访问白名单（CORS）或同源反向代理规则。

  \item \textbf{客户端}\\
  Web 应用，不安装专用桌面软件。用户通过 Chrome、Edge、Firefox 等主流浏览器访问系统。桌面端和笔记本电脑为主要使用设备，平板设备可用于浏览和轻量审批。客户端不要求 GPU、专业 EEG 软件或 MATLAB 环境。
\end{enumerate}

\subsubsection{验收标准}

产品应达到立项时提出的目标，即：

\begin{enumerate}[(1)]
  \item \textbf{完成需求分析书中所列出的各项功能}\\
  系统须通过需求分析报告中定义的 20 项功能验收条件（REQ-001 至 REQ-020），核心验收项包括：
    \begin{itemize}
      \item 用户注册、登录与角色区分（被试者/实验员/管理员），角色过滤导航菜单和数据可见范围。
      \item 实验员创建实验项目、上传 EEG 数据并查看质控报告。
      \item 系统自动生成质控报告，包括格式解析、缺失值比例、异常值数量、通道完整性、采样率校验和综合结论。
      \item 管理员对上传数据进行伦理审核（通过/驳回/冻结），对数据使用申请进行审批授权。
      \item 被试者在线签署和撤回知情同意，查看和撤回本人数据授权。
      \item 经授权用户查看 EEG 波形分析、频谱分析、频段能量分析和 AI 状态识别结果。
      \item 管理员维护系统配置、查看操作审计日志。
      \item 系统提供通知中心和浅色/深色主题切换。
      \item 所有异常场景（无权限、未审核、请求失败、处理失败）均展示明确提示。
    \end{itemize}

  \item \textbf{提交课程要求所列的各项需要提交的文档}\\
  包括《需求分析报告》《项目开发计划书》等全部正式文档，以及答辩演示幻灯片，各文档内容应与系统实际状态一致。

\end{enumerate}
